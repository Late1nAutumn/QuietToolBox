# Isles — infinite canvas module

```
This project is generated by Claude code
```

A self-contained pan / zoom / drag / multi-select / edge-rendering canvas
component for React. Bring your own element shapes and renderer; the module
owns spatial state and stacking.

This README is the complete reference. **An AI agent should be able to make
correct changes to this module without reading the source files** — every
piece of internal state, every event branch, every CSS hook is documented
below. The "Modification recipes" section at the bottom maps common asks to
the exact file and constant to edit.

---

## 1. Public API

### Import

```js
import { Isles } from './Isles';                  // when consuming from sibling
import { Isles } from '../Isles';                 // typical app-side usage
```

Importing the package's `index.js` also injects the module's CSS into
`<head>`. Webpack/vite + `css-loader` is required.

### Component

```jsx
<Isles
  ref                  = {Ref?  /* see §13 for imperative methods */}
  elements             = {CanvasElement[]}                          // required
  onElementsChange     = {(updater | CanvasElement[]) => void}      // required
  renderElement        = {(element, isSelected) => ReactNode}       // required
  edges                = {Edge[]?}
  showGrid             = {boolean?  /* default: true */}
  initialCamera        = {{ x:0, y:0, scale:1 }?}
  virtualize           = {boolean?  /* default: false; see §14 */}
  virtualizeOverscan   = {number?   /* default: 200 (world px) */}
  className            = {string?}
  style                = {CSSProperties?}
/>
```

### Prop semantics

| Prop | Description |
| ---- | ----------- |
| `elements` | The full list of elements. Each has `id`, world-space `x`/`y` (top-left), and optionally `width`/`height`. Extra fields pass through to `renderElement` untouched. |
| `onElementsChange` | Called when a drag changes positions. **Always invoked with a React-style updater fn** (`prev => prev.map(...)`), never with a literal array — your state handler must accept that shape. |
| `renderElement` | Renders the visual body of an element. The returned node is positioned automatically by `<CanvasItem>` (absolute, world-space, hit-target). Do **not** include positioning styles in your rendered node. |
| `edges` | Optional array of connections. Each edge has `source` and `target` (element ids). Edges referencing missing ids are silently skipped. Optional `id` field used as React key; defaults to `"source->target"`. |
| `showGrid` | Toggles the SVG grid background. |
| `initialCamera` | Initial pan + zoom. Subsequent camera changes from user input (wheel zoom, right-drag pan) are owned internally; this prop is read once. To programmatically move the camera after mount, use the imperative ref API in §13. |
| `virtualize` | When true, elements whose AABB is outside the camera viewport (plus `virtualizeOverscan`) are excluded from the DOM. Edges still receive the full element list, so connection lines render correctly even when an endpoint is off-screen. See §14. |
| `virtualizeOverscan` | World-space padding around the viewport used by virtualization (default 200). Larger value = fewer mount/unmount churns during pan, at the cost of more DOM. |
| `className` | Appended to `.isles-root`. Use to scope CSS variable overrides. |
| `style` | Inline-spread on the root container. Prefer CSS overrides via className unless you need dynamic values. |

### `CanvasElement` shape

```ts
{
  id:     string | number      // unique key; identity is preserved across renders
  x:      number               // world-space X of top-left
  y:      number               // world-space Y of top-left
  width:  number?              // optional; falls back to DEFAULT_ELEMENT_WIDTH for hit testing
  height: number?              // optional; falls back to DEFAULT_ELEMENT_HEIGHT
  // any other fields you put here pass through to renderElement & edge endpoints
}
```

**Do not put `zIndex` on elements.** Stacking is owned by the module —
see §6.

### `Edge` shape

```ts
{
  id?:    string               // React key; falls back to "source->target"
  source: elementId            // anchor: right-edge midpoint of this element
  target: elementId            // anchor: left-edge midpoint of this element
}
```

Edges are horizontal cubic-bezier paths (`source.right → target.left`)
with horizontally-offset control points. Anchor positions are derived
from element `x/y/width/height`.

## 2. Interaction model

| Input | Action |
| ----- | ------ |
| Right-click + drag                  | Pan camera |
| Mouse wheel                         | Zoom, anchored to mouse position |
| Left-click element                  | Select (replaces selection) + bring to front |
| Shift + left-click element          | Toggle in selection + bring to front |
| Left-drag from element              | Move all selected elements |
| Left-drag from canvas background    | Rubber-band multi-select (replaces selection) |
| Shift + left-drag canvas background | Rubber-band additive multi-select |
| Mouse release                       | Drag ends; selection committed if it was a rubber-band |

Rubber-band drags shorter than `MIN_RUBBER_BAND_SIZE` (world px) on both
axes are treated as clicks (no selection change). Background click
**replaces** the current selection unless Shift is held.

## 4. State and refs

All in `Isles.jsx`. Refs are used for one of two reasons: (a) avoid
re-renders during drag, or (b) keep `window`-mounted listeners reading the
latest values without re-binding.

### State

| Name | Shape | Trigger to change |
| ---- | ----- | ----------------- |
| `selectedIds`   | `Set<elementId>`                                  | Element mousedown (replace/toggle), rubber-band release, background mousedown without Shift (clears) |
| `selectionRect` | `{x,y,width,height}` in world coords, or `null`   | Set during rubber-band drag (mousemove). Cleared on mouseup. |
| `order`         | `elementId[]` — later = on top                    | Reconciled with `elements` whenever its id-set changes; mutated by element mousedown (selected ids → end). |
| `camera`        | `{x, y, scale}` (owned by `useCamera`)            | Mouse wheel (zoomAt), right-drag (pan), or the imperative ref API (§13). |
| `containerSize` | `{width, height}` (only updated when `virtualize` is on) | ResizeObserver on the root container. Drives viewport culling. |

### Refs

| Name | Holds | Why |
| ---- | ----- | --- |
| `containerRef`        | The root DOM node             | Needed for `getBoundingClientRect()` in coord conversion. |
| `cameraRef`           | Latest `camera`               | Read during mousemove without rebinding. |
| `elementsRef`         | Latest `elements`             | Used at rubber-band release to hit-test. |
| `onElementsChangeRef` | Latest `onElementsChange` cb  | Called during MOVING drag. |
| `selectedIdsRef`      | Latest `selectedIds`          | Read in `handleElementMouseDown` to compute next selection synchronously (avoids React 18 batching race — see §5). |
| `dragRef`             | `{type, ...payload}`          | The whole drag state machine. Mutated freely on mousedown/move/up. |

### `dragRef` shapes per state

```js
{ type: DRAG_TYPE.IDLE }
{ type: DRAG_TYPE.PANNING,   lastX, lastY }
{ type: DRAG_TYPE.SELECTING, startWorld, additive: boolean, currentWorld? }
{ type: DRAG_TYPE.MOVING,    lastX, lastY, movingIds: Set<id> }
```

## 5. Event flow

Four input points:

1. **`containerRef` onWheel** — preventDefault + `zoomAt`. Camera updates.
2. **`containerRef` onMouseDown** (canvas background) — sets up PANNING (right) or SELECTING (left).
3. **Each `.isles-item` onMouseDown** (element) — sets up MOVING drag, updates selection + bring-to-front. `stopPropagation()` keeps the event from reaching the canvas-background handler above.
4. **`window` onMouseMove / onMouseUp** — single global pair, mounted once, dispatches on `dragRef.current.type`. Uses refs to read latest values without re-binding.

### Critical detail: React 18 batching + `dragRef`

`handleElementMouseDown` must hand `dragRef` the same `movingIds` Set that it gives `setSelectedIds`. If it wrote:

```js
let next;
setSelectedIds(prev => { next = compute(prev); return next; });
dragRef.current = { movingIds: next };  // BUG: updater hasn't run yet
```

…the updater runs **later** (React 18 batches), so `next` is still
`undefined` when `dragRef` is assigned, and the drag would move the
previous selection (or nothing). The current code **computes `next`
synchronously from `selectedIdsRef.current`**, then calls
`setSelectedIds(next)` and assigns `dragRef.movingIds = next` from the
same value. Do not revert this without re-introducing the race.

### mousemove dispatch

```
PANNING   → call useCamera.pan(dx, dy)
SELECTING → update selectionRect (world-space marquee)
MOVING    → call onElementsChange(prev => prev.map(applyDelta))
```

### mouseup dispatch

- `SELECTING` → build final marquee rect, hit-test every element via `rectsOverlap`, replace/extend `selectedIds`. Skip if marquee is below `MIN_RUBBER_BAND_SIZE` on both axes.
- All other types → reset to IDLE.

## 6. Stacking model

Z-order is **internal**. Elements have no `zIndex` field.

The module maintains `order: string[]` — an array of element ids where
**later = visually higher**. At render time:

```
sortedElements = elements sorted by orderIndex(id)        // back-to-front
each CanvasItem receives zIndex = orderIndex(id)
```

### Reconciliation with `elements` prop

An effect compares the id-set of `elements` with `order`:

- Same set → returns the same `order` reference (React skips re-render).
- Different → filter out vanished ids, append new ids to the end (new cards sit on top).

### Bring-to-front trigger

Element mousedown moves all soon-to-move ids (= the just-computed next
selection) to the end of `order`, preserving their relative order. So a
Shift-click adding an element to a multi-selection keeps the
already-selected ones together at the top.

### Edge layer

`.isles-edge-svg` has `z-index: 100`, which sits above `.isles-world`
(z-index: auto). Therefore **connection lines are always drawn above all
cards** (and below the rubber-band rect at `z-index: 9999`). This is a
deliberate design choice — change `z-index: 100` in `Isles.css` to flip.

## 7. Coordinate system

Three spaces:

- **Screen (client)** — what `MouseEvent.clientX/Y` reports
- **Container-local** — screen minus `containerRef`'s bounding rect
- **World** — what element `x/y` and `selectionRect` live in

Conversions (the module's `toLocal` and `toWorld` helpers):

```
local.x  = clientX - rect.left
world.x  = (local.x - camera.x) / camera.scale
screen.x = world.x * camera.scale + camera.x       // inverse, used by SelectionRect & EdgeLayer
```

`.isles-world` applies `transform: translate(camera.x, camera.y) scale(camera.scale)`
to render the world layer in screen space. SVG layers (grid, edges)
apply the same transform inside a `<g>` so geometry is authored in world
units. The grid SVG additionally adjusts its `pattern` x/y offset so
content scrolls with pan.

## 8. Theming — CSS variables

Defaults are supplied as inline `var(name, fallback)` fallbacks at each
use site — they are **not** set on `.isles-root` itself. This matters:
if defaults were set on `.isles-root`, an ancestor's same-name override
would lose the cascade (an element's own declaration beats anything
inherited). With fallbacks, the cascade resolves to the nearest ancestor
that defines the variable, and the inline fallback only kicks in if
nobody up the tree defines it. So:

```css
/* This works — ancestor override is respected */
.my-app .isles-root,
.my-app {
  --isles-bg: #1a1a1a;
}
```

Multiple Isles instances on the same page can still be themed
independently — either by attaching different ancestor classes, or by
setting the var on the `.isles-root` element directly (which always
wins since it's on the element itself).

| Variable                    | Default                     | Purpose                                  |
| --------------------------- | --------------------------- | ---------------------------------------- |
| `--isles-bg`                | `#fafafa`                   | Canvas background                        |
| `--isles-grid-color`        | `#e2e2e2`                   | Fine grid line                           |
| `--isles-grid-accent-color` | `#c8c8c8`                   | Every-Nth accent line                    |
| `--isles-selection-color`   | `#4a90e2`                   | Selected outline + rubber-band border    |
| `--isles-selection-fill`    | `rgba(74,144,226,0.08)`     | Rubber-band fill                         |
| `--isles-edge-color`        | `#888`                      | Connection-line stroke                   |

Override example:

```css
.my-app .isles-root {
  --isles-bg: #1a1a1a;
  --isles-grid-color: #2a2a2a;
}
```

| Name                       | Default | Role                                                                                  |

| `MIN_SCALE`                | `0.05`  | Zoom-out floor                                                                        |

## 11. Modification recipes

| You want to…                              | Edit                                                                                                   |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Move the camera from outside              | Use the imperative ref API — `setCamera`, `centerOn`, `fitToContent`. See §13                          |
| Render 1k+ elements without DOM bloat     | Set `virtualize` to true. See §14                                                                      |
| Change zoom speed                         | `ZOOM_STEP` in `constant.js`                                                                           |
| Change zoom range                         | `MIN_SCALE` / `MAX_SCALE` in `constant.js`                                                             |
| Change grid density / accent spacing      | `GRID_BASE_SIZE`, `GRID_MIN_PX`, `GRID_ACCENT_MULTIPLIER` in `constant.js`                            |
| Change edge thickness                     | `EDGE_STROKE_WIDTH` in `constant.js`                                                                   |
| Change edge curvature                     | `EDGE_CONTROL_OFFSET_MIN` in `constant.js`, or rewrite the `d="..."` line in `EdgeLayer.jsx`           |
| Put edges below cards                     | Lower `.isles-edge-svg`'s z-index in `Isles.css` to e.g. `0`. See §6.                                  |
| Recolor anything                          | Override the relevant `--isles-*` variable in consumer CSS — do not edit `Isles.css`                   |
| Add a third drag mode (e.g. mid-mouse)    | Add a `DRAG_TYPE` member; new branches in `handleMouseDown` and in the global `onMove`/`onUp`          |
| Externalize stacking                      | `order` is `useState` in `Isles.jsx`. Lift to controlled-prop pattern: accept `order`/`onOrderChange`, derive when provided, fall back to internal state |
| Hit-test with arbitrary shapes            | Replace `rectsOverlap(selRect, elRect)` in the SELECTING mouseup branch of `Isles.jsx`                 |
| Disable selection entirely                | Skip `setSelectedIds` calls in `handleMouseDown` and `handleElementMouseDown`, and gate the SELECTING branch on a prop |
| Disable element drag                      | In `handleElementMouseDown`, don't write `dragRef.current = { type: MOVING, ... }`                     |
| Change cursor while panning               | Currently no panning cursor. Adding one means tracking active drag type as state (re-renders on type change, fine — once per drag boundary) |

## 13. Imperative API (ref)

Camera changes from user input (wheel/pan) are owned internally. To move
the camera programmatically — e.g., re-center after a layout pass, or
fit-to-content on demand — attach a ref and call the methods exposed
via `forwardRef` + `useImperativeHandle`.

```jsx
const islesRef = useRef(null);

<Isles ref={islesRef} ... />

// Get the current camera
const cam = islesRef.current.getCamera();          // { x, y, scale }

// Set the camera directly (raw)
islesRef.current.setCamera({ x: 0, y: 0, scale: 1 });

// Pan so that a world point lands at the container center
islesRef.current.centerOn({ x: 300, y: 200 });          // keeps current scale
islesRef.current.centerOn({ x: 300, y: 200, scale: 1.5 }); // also sets scale

// Frame a set of elements (or all current elements) with padding
islesRef.current.fitToContent();                       // uses props.elements
islesRef.current.fitToContent(someSubset, 120);        // 120 world-px padding
```

| Method | Signature | Behavior |
| ------ | --------- | -------- |
| `getCamera()`     | `() => { x, y, scale }`                                 | Returns the current camera object (read from the live ref, not the last render snapshot). |
| `setCamera(c)`    | `({ x, y, scale }) => void`                             | Replaces the camera unconditionally. No clamping beyond what `useCamera`'s setter enforces (= none). |
| `centerOn(p)`     | `({ x, y, scale? }) => void`                            | Pans so world point `(x, y)` lands at the container's center. Reads container size from `getBoundingClientRect()`. |
| `fitToContent(els?, padding?)` | `(CanvasElement[]?, number?) => void`           | Computes the AABB of `els` (defaults to `props.elements`), then sets a camera that places that AABB in the container minus `padding` world-px (default `80`). Scale is clamped to `[MIN_SCALE, MAX_SCALE]`. |

Implementation lives in `Isles.jsx`'s `useImperativeHandle(...)`. All
methods short-circuit silently if the container isn't mounted yet or
the input list is empty.

## 14. Virtualization (`virtualize` prop)

When `virtualize={true}`, only elements whose AABB intersects the
camera viewport (expanded by `virtualizeOverscan` world-px on each
side) are rendered into the DOM. The rest still exist in
`props.elements` and remain addressable — they're just not mounted as
`.isles-item` nodes.

### Why this is correct for edges

`EdgeLayer` receives the **full** `props.elements` array, not the
culled list. So a `<path>` whose endpoint is off-screen still resolves
to valid coordinates and draws correctly across the viewport. Only the
endpoint card's DOM is omitted.

### Cost model

- A `ResizeObserver` watches the container and updates a
  `containerSize` state whenever it changes. This is the only
  virtualization-specific side effect.
- Each camera change re-runs the cull filter inside a `useMemo`. The
  filter is `O(elements)` per frame; for thousands of elements, the
  win comes from React reconciling a much smaller children array and
  the browser laying out fewer DOM nodes.
- `virtualizeOverscan` trades DOM count for mount/unmount churn during
  fast panning. Bump it if you see flicker; lower it if mount count is
  still too high.

### When to leave it off (the default)

For small canvases (≲200 elements with simple `renderElement` output),
virtualization adds a memo, a `ResizeObserver`, and an extra
camera-dep render path for no real win. Keep it off.

### Limitations

- No virtualization on the SELECTING hit test — rubber-band selection
  iterates the full `elementsRef.current` (correct: a marquee may
  intersect off-screen elements).
- No virtualization on edges. Path count = full edge count. Add a
  pre-filter in `EdgeLayer` if your edge count dwarfs your element
  count.
- The `order` state still tracks all element ids (so stacking remains
  stable across viewport changes).
